## 1. 产品概述

### 1.1 产品定位

API 编排平台是一个**面向企业客户的私有化部署接口编排内部工具**。产品以 API 接口编排为核心能力，帮助企业技术团队快速连接外部系统、定义可调用接口、可视化编排业务流程、自动化执行与监控。

### 1.2 核心价值主张

| 维度 | 价值 |
|------|------|
| **效率提升** | 可视化编排替代代码编写，非技术人员也能配置接口集成流程 |
| **降低门槛** | 批量导入 OpenAPI/proto/sqlc，无需从零定义每个接口 |
| **流程自动化** | 支持定时任务、消息队列、API 调用等多种触发方式 |
| **企业级可靠性** | 私有化部署、多租户隔离、RBAC 权限、审计日志 |
| **可扩展性** | 插件化架构，支持自定义资源类型和触发器类型 |

### 1.3 产品边界

**在范围内：**
- 外部服务连接管理（HTTP/gRPC/数据库等）
- 接口工具定义与管理（单个创建 + 批量导入）
- 可视化流程编排画布
- 流程触发器配置（RESTful/定时器/MQ）
- 运行记录与监控
- 多租户工作空间与 RBAC 权限

**不在范围内：**
- 接口 Mock 服务（可后续扩展）
- SLA/告警平台集成（MVP 后续迭代）
- API 文档门户（MVP 后续迭代）
- 流程编排性能压测（MVP 后续迭代）

### 1.4 核心数据模型

平台采用四层递进数据模型：

```
资源 (Resource) → 工具 (Tool) → 流程 (Flow) → 触发器 (Trigger)
```

| 层级 | 说明 | 类比 |
|------|------|------|
| **资源** | 外部系统的连接配置（地址、认证等） | 数据库连接串 |
| **工具** | 资源暴露的可调用能力（单个接口/方法/SQL） | SQL 语句 / API Endpoint |
| **流程** | 工具的编排组合（调用顺序、数据映射） | 工作流 Pipeline |
| **触发器** | 流程的外部入口（输入输出映射） | API Route / Cron Job |

**层级依赖关系：**
- 工具归属于某个资源
- 流程编排已发布的工具
- 触发器绑定已发布的流程
- 每一层独立版本管理，上层锁定下层具体版本

## 3. 系统架构概览

### 3.1 部署架构

平台采用**控制台(Console) + 执行器(Executor)** 双角色架构，两者编译到同一二进制，通过 CLI 参数区分角色：

```
┌─────────────────────────────────────────────────────────────────┐
│                        Console (控制台)                          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────────┐  │
│  │   Web UI    │  │   API 服务   │  │      发布服务            │  │
│  │  (React)    │  │   (REST)    │  │  (同步配置到 etcd)       │  │
│  └─────────────┘  └─────────────┘  └─────────────────────────┘  │
│                          │                                      │
│                          ▼                                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │              数据库 (PostgreSQL)                         │   │
│  │  - 资源/工具/流程/触发器定义                              │   │
│  │  - 工作空间/用户/权限                                    │   │
│  │  - 运行记录/审计日志                                     │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼ 发布配置
┌─────────────────────────────────────────────────────────────────┐
│                        etcd 配置中心                             │
│  /clusters/{cluster-id}/                                        │
│    ├── /flows/{flow-id}          # 流程定义                      │
│    ├── /triggers/{trigger-id}    # 触发器配置                    │
│    ├── /resources/{res-id}       # 资源配置                      │
│    └── /tools/{tool-id}          # 工具定义                      │
└─────────────────────────────────────────────────────────────────┘
                                │
              ┌─────────────────┼─────────────────┐
              ▼                 ▼                 ▼
┌─────────────────┐   ┌─────────────────┐   ┌─────────────────┐
│  Cluster A      │   │  Cluster B      │   │  Cluster C      │
│  (env: dev)     │   │  (env: staging) │   │  (env: prod)    │
│                 │   │                 │   │                 │
│  ┌───────────┐  │   │  ┌───────────┐  │   │  ┌───────────┐  │
│  │ Executor  │  │   │  │ Executor  │  │   │  │ Executor  │  │
│  │  x N      │  │   │  │  x N      │  │   │  │  x N      │  │
│  └───────────┘  │   │  └───────────┘  │   │  └───────────┘  │
└─────────────────┘   └─────────────────┘   └─────────────────┘
```

### 3.2 角色说明

| 角色 | 启动方式 | 职责 | 数据存储 |
|------|----------|------|----------|
| **Console** | `./app --role=console` | 提供 Web UI 和 API；管理资源/工具/流程/触发器定义；向 etcd 发布配置 | PostgreSQL |
| **Executor** | `./app --role=executor --etcd-endpoints=... --cluster-id=...` | 连接 etcd 监听配置变更；执行触发器和流程；上报运行状态 | 无本地状态 |

### 3.3 核心概念

#### 环境 (Environment)
- **定义**：对集群的标签/分类，用于逻辑分组
- **示例**：`dev`、`staging`、`production`
- **作用**：帮助用户按环境维度管理和筛选集群
- **关系**：一个环境可包含多个集群（一对多）

#### 集群 (Cluster)
- **定义**：一组运行相同配置的执行器实例
- **关键属性**：
  - 唯一标识 `cluster_id`
  - 所属环境 `environment`
  - 绑定的 etcd 端点 `etcd_endpoints`
  - etcd 中的根节点路径 `/clusters/{cluster_id}`
- **关系**：
  - 一个集群绑定**唯一一套 etcd**（一对多）
  - 一套 etcd 可服务**多个集群**（通过不同根节点区分）

#### 配置中心 (etcd)
- **作用**：Console 与 Executor 之间的配置同步桥梁
- **存储结构**：
  ```
  /clusters/{cluster-id}/
    ├── flows/{flow-id}          → 流程定义 JSON
    ├── triggers/{trigger-id}    → 触发器配置 JSON
    ├── resources/{res-id}       → 资源配置 JSON
    └── tools/{tool-id}          → 工具定义 JSON
  ```
- **同步机制**：
  - Console 发布 → 写入 etcd
  - Executor 监听 → 通过 Watch API 实时同步

### 3.4 发布流程

```
用户在 Console 中操作：
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 编辑流程 │ → │ 测试验证 │ → │ 选择集群 │ → │ 发布    │
└─────────┘    └─────────┘    └─────────┘    └────┬────┘
                                                  │
                    写入 etcd /clusters/{cluster-id}/flows/{flow-id}
                                                  │
                                                  ▼
┌─────────────────────────────────────────────────────────────────┐
│                     Executor (集群内所有实例)                      │
│  监听 etcd Watch → 配置变更 → 热加载 → 新流程生效                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.5 执行器启动与配置获取

```bash
# 执行器启动示例
./app --role=executor \
      --etcd-endpoints=etcd-1:2379,etcd-2:2379,etcd-3:2379 \
      --cluster-id=cluster-prod-001 \
      --log-level=info
```

启动流程：
1. 解析 CLI 参数，确定角色为 Executor
2. 连接指定 etcd 集群
3. 监听 `/clusters/{cluster-id}/` 前缀的所有键
4. 全量拉取当前配置（触发器、流程、资源、工具）
5. 启动触发器（RESTful Server / Cron Scheduler / MQ Consumer）
6. 进入 Watch 循环，实时响应配置变更

## 2. 目标用户与使用场景

### 2.1 用户画像

#### 主要用户：接口开发者 / 集成工程师

| 维度 | 描述 |
|------|------|
| **背景** | 2-5 年开发经验，熟悉 HTTP、JSON、RESTful API |
| **目标** | 快速创建和编排接口，完成业务系统集成 |
| **痛点** | 创建→编辑→测试需要反复跳转页面，上下文丢失 |
| **工具经验** | 日常使用 Postman、GitLab CI/CD、Jira 等开发者工具 |
| **核心诉求** | 效率优先、流程连贯、操作可回溯、错误可快速定位 |

#### 次要用户：系统管理员

| 维度 | 描述 |
|------|------|
| **背景** | 负责平台运维、权限管理、环境配置 |
| **目标** | 管理工作空间成员、监控运行状态、确保系统稳定 |
| **核心诉求** | 权限控制清晰、运行日志可审计、系统健康可感知 |

#### 潜在用户：业务人员（低代码使用者）

| 维度 | 描述 |
|------|------|
| **背景** | 了解业务逻辑，但不一定具备编程能力 |
| **目标** | 配置简单的接口集成流程，减少对开发人员的依赖 |
| **核心诉求** | 操作简单、可视化、有模板参考 |

### 2.2 核心使用场景

#### 场景一：连接外部系统并定义接口（资源 → 工具）

> 作为开发者，我需要将公司的用户服务（HTTP）、支付网关（gRPC）、订单数据库（MySQL）接入平台，定义可调用的接口工具。

**操作路径：** 创建资源 → 测试连接 → 批量导入工具（OpenAPI 文件）→ 预览确认 → 发布

**成功标准：** 5 分钟内完成一个 HTTP 服务的全部工具导入并发布

#### 场景二：编排用户注册流程（流程编排）

> 作为开发者，我需要在用户注册时依次调用创建用户、发送欢迎邮件、记录注册日志三个接口。

**操作路径：** 创建流程 → 拖拽工具节点到画布 → 配置参数映射 → 测试运行 → 发布

**成功标准：** 可视化完成三节点串行编排，测试运行通过

#### 场景三：暴露注册接口供外部调用（触发器）

> 作为开发者，我需要将用户注册流程暴露为 RESTful API，供前端或第三方系统调用。

**操作路径：** 创建触发器 → 选择 RESTful 类型 → 绑定已发布流程 → 配置输入输出映射 → 启用

**成功标准：** 外部系统可 POST 调用注册接口，正确返回注册结果

#### 场景四：定时执行日终对账（定时触发器）

> 作为开发者，我需要每天凌晨 2 点自动执行对账流程。

**操作路径：** 创建触发器 → 选择定时器类型 → 配置 Cron 表达式 → 绑定对账流程 → 启用

**成功标准：** 按计划自动执行，运行记录可追溯

#### 场景五：排查看运行失败原因（运行监控）

> 作为开发者，我需要查看最近一次失败的运行记录，定位是哪个节点报错。

**操作路径：** 运行记录列表 → 筛选失败记录 → 查看详情 → 节点时间线 → 展开失败节点 → 查看错误信息

**成功标准：** 30 秒内定位到失败节点和错误原因

## 4. 核心数据模型

### 4.1 模型关系图

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                              平台核心模型                                         │
│                                                                                 │
│   ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐  │
│   │  环境     │    │  集群     │    │  资源     │    │  工具     │    │  流程     │  │
│   │Environment│   │ Cluster  │    │ Resource │───→│  Tool    │───→│  Flow    │  │
│   └────┬─────┘    └────┬─────┘    └──────────┘    └──────────┘    └────┬─────┘  │
│        │               │                                              │         │
│        │    ┌──────────┘                                              │         │
│        │    │                                                         ▼         │
│        │    │    ┌──────────────────────────────────────────────────────────┐   │
│        │    └───→│                      触发器                              │   │
│        │         │                     Trigger                              │   │
│        │         │  (绑定集群：同一流程可在不同集群有不同的触发器配置)         │   │
│        │         └──────────────────────────────────────────────────────────┘   │
│        │                                                                        │
│   ┌────┴────┐    ┌──────────┐    ┌──────────┐                                  │
│   │ 工作空间  │    │  用户     │    │  角色     │                                  │
│   │Workspace│───→│  User    │───→│  Role    │                                  │
│   └─────────┘    └──────────┘    └──────────┘                                  │
└─────────────────────────────────────────────────────────────────────────────────┘
```

**关键关系说明：**
- **环境**是集群的标签，一个环境可包含多个集群
- **集群**绑定一套 etcd，资源/工具/流程定义按工作空间管理，但**触发器按集群部署**
- 同一流程可以在不同集群配置不同的触发器（如 dev 集群用测试 MQ，prod 集群用生产 MQ）

### 4.2 环境 (Environment)

**定义：** 对集群的分类标签，用于逻辑分组管理。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | UUID | 自动 | 主键 |
| name | string(50) | ✅ | 环境名称，如 "开发环境"、"生产环境" |
| key | string(20) | ✅ | 环境标识，如 `dev`、`staging`、`prod`，唯一 |
| description | string(200) | ❌ | 环境描述 |
| color | string(7) | ❌ | 环境标签颜色（Hex 格式，如 `#10B981`） |
| created_at | timestamp | 自动 | 创建时间 |
| updated_at | timestamp | 自动 | 更新时间 |

**预设环境：**
- `dev` - 开发环境（绿色）
- `staging` - 预发布环境（黄色）
- `prod` - 生产环境（红色）

### 4.3 集群 (Cluster)

**定义：** 一组运行相同配置的执行器实例，绑定唯一的 etcd 配置中心。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | UUID | 自动 | 主键 |
| name | string(100) | ✅ | 集群名称 |
| environment_id | UUID | ✅ | 所属环境 |
| etcd_endpoints | string[] | ✅ | etcd 端点列表，如 `["etcd-1:2379", "etcd-2:2379"]` |
| etcd_username | string | ❌ | etcd 认证用户名 |
| etcd_password | string | ❌ | etcd 认证密码（加密存储） |
| etcd_prefix | string | 自动 | etcd 中的根节点前缀，默认 `/clusters/{cluster-id}` |
| description | string(500) | ❌ | 集群描述 |
| status | enum | 自动 | 状态：active / inactive / error |
| executor_count | int | 自动 | 当前在线执行器数量（心跳统计） |
| last_heartbeat | timestamp | 自动 | 最后心跳时间 |
| created_at | timestamp | 自动 | 创建时间 |
| updated_at | timestamp | 自动 | 更新时间 |

**集群状态流转：**
```
创建 → active → inactive（手动停用）
              ↓
            error（etcd 连接异常）
```

### 4.4 资源 (Resource)

**定义：** 外部系统（HTTP 服务、gRPC 服务、数据库等）的连接配置。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | UUID | 自动 | 主键 |
| name | string(100) | ✅ | 资源名称 |
| type | enum | ✅ | 资源类型：http / grpc / mysql / postgres / ...（插件扩展） |
| config | JSON | ✅ | 连接配置（结构由插件 Schema 定义） |
| workspace_id | UUID | ✅ | 所属工作空间 |
| status | enum | 自动 | 连接状态：active / inactive / error |
| description | string(500) | ❌ | 资源描述 |
| created_at | timestamp | 自动 | 创建时间 |
| updated_at | timestamp | 自动 | 更新时间 |

**资源类型配置 Schema：**

| 资源类型 | 必填配置项 | 可选配置项 |
|---------|-----------|-----------|
| HTTP | baseUrl, auth.type | auth.token, auth.apiKey, auth.username, auth.password, timeout, headers, TLS 配置 |
| gRPC | serverAddress, protoDefinition | auth, timeout, TLS 配置 |
| MySQL | host, port, database, username, password | charset, maxOpenConns, maxIdleConns, timeout |
| PostgreSQL | host, port, database, username, password | sslMode, maxOpenConns, timeout |

### 4.5 工具 (Tool)

**定义：** 资源暴露的可调用能力。每个工具代表一个具体的接口/方法/SQL 操作。

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | UUID | 自动 | 主键 |
| name | string(100) | ✅ | 工具名称 |
| resource_id | UUID | ✅ | 所属资源 |
| type | enum | ✅ | 工具类型：http_method / grpc_method / sql_query / ... |

### 4.7 触发器 (Trigger)

**定义：** 流程的外部入口，负责接收外部请求并转发给流程执行，处理输入输出映射。**触发器按集群部署**，同一流程可以在不同集群配置不同的触发器。

**触发器与集群的关系：**
- 触发器定义在工作空间层面（模板）
- 触发器实例按集群部署（每个集群独立配置）
- 发布时，用户选择目标集群，触发器配置写入该集群的 etcd

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | UUID | 自动 | 主键 |
| name | string(100) | ✅ | 触发器名称 |
| type | enum | ✅ | 类型：restful / timer / mq / ...（插件扩展） |
| config | JSON | ✅ | 触发器配置（结构由插件 Schema 定义） |
| flow_id | UUID | ✅ | 绑定的流程 |
| flow_version | int | ✅ | 锁定的流程版本 |
| cluster_id | UUID | ✅ | **部署目标集群** |
| input_mapping | JSON | ✅ | 输入映射：外部请求 → 流程输入参数 |
| output_mapping | JSON | ✅ | 输出映射：流程返回值 → 外部响应 |
| error_handling | JSON | ✅ | 错误处理配置 |
| workspace_id | UUID | ✅ | 所属工作空间 |
| status | enum | ✅ | 状态：active / paused |
| created_at | timestamp | 自动 | 创建时间 |
| updated_at | timestamp | 自动 | 更新时间 |

**触发器类型配置：**

| 类型 | 必填配置 | 输入映射 | 输出映射 |
|------|---------|---------|---------|
| RESTful | method, path | 请求体/Header/Query → 流程输入 | 流程返回值 → HTTP 响应体/状态码 |
| Timer | cron_expression | 系统注入（触发时间戳、执行编号）→ 流程输入 | 无（结果记录到运行日志） |
| MQ | topic, consumer_group, broker_config | 消息体/Header → 流程输入 | 流程返回值 → 回复消息/转发 Topic |

**错误处理配置：**

```json
{
  "strategy": "custom",
  "customErrorFormat": {
    "code": "{{error.statusCode}}",
    "message": "{{error.message}}",
    "traceId": "{{run.id}}"
  },
  "retryOnFailure": false,
  "fallbackResponse": null
}
```

## 17. 版本与依赖管理策略

### 17.1 版本锁定

- 流程创建时**锁定**引用工具的具体版本（如 v2），运行时始终使用该版本
- 触发器绑定流程时**锁定**流程版本，流程重新发布后需手动指向新版本

### 17.2 工具版本升级

| 步骤 | 说明 |
|------|------|
| 1. 工具发布新版本 | 自动检测下游依赖流程列表 |
| 2. 影响分析 | 展示依赖流程及其工作空间归属 + 接口变更对比（Breaking/Non-breaking） |
| 3. 通知下游 | 依赖旧版本的流程在编排画布中标记 🔄 |
| 4. 手动升级 | 开发者在编排画布中查看新旧版本变更对比，选择是否升级 |
| 5. 兼容性检查 | 升级后自动检测参数映射兼容性，不兼容时提示手动调整 |
| 6. 支持批量 | 支持单个节点升级或批量升级 |

### 17.3 下线保护

| 操作 | 保护机制 |
|------|---------|
| 工具下线 | 有流程引用（任何版本）的工具不允许下线，展示依赖流程列表 |
| 资源下线 | 检查其下所有工具的下游依赖，汇总展示完整影响链路 |
| 流程下线 | 检查是否有触发器绑定，有绑定则阻止下线 |

### 17.4 版本管理

- 版本号格式：**3段版本号**（x.y.z），如 v1.0.0、v1.1.0、v2.0.0
- 每次发布自动递增版本号（v1.0.0 → v1.0.1 → v1.1.0 → v2.0.0）
- 支持查看版本历史列表
- 支持版本对比（Diff）
- 支持版本回滚（回滚生成新版本号，非原地覆盖）
- 版本快照存储完整的配置数据

## 19. 里程碑规划

### 里程碑 1：MVP（最小可行产品）

**目标：** 完成核心链路 — 资源 → 工具 → 流程 → 触发器 → 运行

**交付范围：**
- ✅ 登录 / 认证（本地账号 + SSO）
- ✅ 资源管理（HTTP/gRPC/MySQL，连接测试）
- ✅ 工具管理（单个创建 + OpenAPI 批量导入 + 测试 + 发布/下线）
- ✅ 流程编排画布（节点拖拽/连线/参数映射/测试运行/发布）
- ✅ 触发器管理（RESTful/Timer，输入输出映射）
- ✅ 运行记录（列表 + 详情 + 节点时间线）
- ✅ 工作台 Dashboard
- ✅ 多租户工作空间 + RBAC 权限
- ✅ 深色/浅色主题切换

**预估工期：** 8-10 周

### 里程碑 2：增强与完善

**目标：** 提升体验完整性和运维能力

**交付范围：**

- ✅ 下线保护（完整依赖链路检查）
- ✅ 个人中心（API Keys、偏好设置）
- ✅ 用户管理（平台管理员）
- ✅ 插件管理页面
- ✅ 系统通知与告警

**预估工期：** 4-6 周

### 里程碑 3：高级功能

**目标：** 差异化竞争力和规模化能力

**交付范围：**
- ✅ 条件分支节点、循环节点、子流程
- ✅ 可视化数据映射（拖拽连线式）
- ✅ 自定义角色与权限矩阵
- ✅ 国际化（i18n）
