# 用户旅程地图 — API 编排平台

> 产品定位：面向企业客户的私有化部署 API 编排平台（多租户、工作空间协作）
> 目标用户：内部技术/业务人员（接口开发者、集成工程师、系统管理员）
> 核心数据模型：**资源(Resource) → 工具(Tool) → 流程(Flow) → 触发器(Trigger)**
> 核心能力：外部资源连接 → 工具定义与管理 → 可视化流程编排 → 触发器配置 → 运行监控

---

## 1. 用户画像

### 👤 主要用户：接口开发者 / 集成工程师

| 维度 | 描述 |
|------|------|
| **背景** | 有 2-5 年开发经验，熟悉 HTTP 协议、JSON、RESTful API |
| **目标** | 快速创建和编排接口，完成业务系统集成 |
| **痛点** | 当前工具流程断裂：创建→编辑→测试需要反复跳转页面，上下文丢失 |
| **工具经验** | 日常使用 Postman、GitLab CI/CD、Jira 等开发者工具 |
| **核心需求** | 效率优先、流程连贯、操作可回溯、错误可快速定位 |

### 👤 次要用户：系统管理员

| 维度 | 描述 |
|------|------|
| **背景** | 负责平台运维、权限管理、环境配置 |
| **目标** | 管理工作空间成员、监控运行状态、确保系统稳定 |
| **核心需求** | 权限控制清晰、运行日志可审计、系统健康可感知 |

## 3. 详细旅程地图

### 阶段一：首次登录 & 引导（Onboarding）

| 维度 | 内容 |
|------|------|
| **用户行为** | 登录系统 → 查看工作台 → 了解产品功能 |
| **接触点** | 登录页 → 工作台 Dashboard → 引导提示 |
| **情绪曲线** | 😐 中立 → 🙂 有期待 → 😊 清晰 |
| **用户想法** | "这个系统能帮我做什么？" "从哪里开始？" |
| **当前痛点** | 新用户面对空白画布无从下手 |
| **⚠️ 断点识别** | **引导缺失**：用户不知道系统的核心功能和操作路径 |

**优化方案：**
- 首次登录展示交互式引导（2-3 步 Tooltip 引导核心功能入口）
- 工作台提供"快速开始"模板：一键创建示例编排流程
- 左侧导航高亮当前可操作的功能模块

---

### 阶段二：创建资源（Resource Setup）

| 维度 | 内容 |
|------|------|
| **用户行为** | 选择资源类型（HTTP/gRPC/DB/插件）→ 填写连接配置 → 测试连接 → 创建 |
| **接触点** | 资源列表 → 创建资源页（类型选择 + 连接配置） |
| **情绪曲线** | 🙂 明确目标 → 😐 填写配置 → 😟 担心配置错误 → 😊 连接测试成功 |
| **用户想法** | "这个服务地址对吗？" "认证信息怎么填？" "能连通吗？" |
| **当前痛点** | 不同资源类型的配置表单差异大，初次配置容易出错 |
| **⚠️ 断点识别** | **配置校验不即时 + 错误提示不明确** |

**优化方案：**
- **类型感知表单**：选择资源类型后，动态展示对应配置字段 + 字段说明
- **即时校验**：URL 格式、端口范围等字段实时校验
- **连接测试**：必填完成后启用"测试连接"按钮，即时反馈连通性
- **配置模板**：常用配置（如 Bearer Token）提供预设模板

---

### 阶段三：创建 & 导入工具（Tool Setup）

| 维度 | 内容 |
|------|------|
| **用户行为** | 单个创建工具 / 批量导入（OpenAPI/proto/sqlc）→ 配置输入输出参数 → 测试 → 发布 |
| **接触点** | 工具列表 / 工具编辑器 / 批量导入向导 |
| **情绪曲线** | 🙂 明确目标 → 😐 配置参数 → 😟 参数较多 → 😊 测试通过 → 😐 发布流程 |
| **用户想法** | "手动创建还是批量导入？" "导入后工具对不对？" "这个工具可以发布了吗？" |
| **当前痛点** | 批量导入预览不够清晰；发布状态不可见；已发布工具的版本管理困难 |
| **⚠️ 断点识别** | **导入预览不清晰 + 发布状态不可见 + 发布后才可在流程中使用** |

**优化方案：**
- **批量导入预览**：三步向导（选择方式 → 上传文件 → **逐条预览 + 勾选确认**）
- **发布状态徽章**：工具列表每行显示 🟡草稿 / ✅已发布 / ⬇已下线
- **发布前置检查**：发布前自动检查输入输出参数完整性
- **版本管理**：每次发布自动生成版本号，支持版本对比和回滚

---

### 阶段四：创建编排流程（Orchestrate）

| 维度 | 内容 |
|------|------|
| **用户行为** | 创建新编排流程 → 进入编排画布 → 从节点面板拖拽接口节点到画布 |
| **接触点** | 编排列表 → 创建流程 → 编排画布（空画布 + 左侧节点面板） |
| **情绪曲线** | 🙂 明确目标 → 😐 面对空白画布 → 😟 不确定下一步 |
| **用户想法** | "怎么开始？先加哪个节点？" "这个画布怎么操作？" |
| **当前痛点** | 空画布无引导、画布操作（缩放/平移）不直观 |
| **⚠️ 断点识别** | **空白画布综合征 + 操作可发现性低** |

**优化方案：**
- **画布引导层**：首次进入画布时显示"拖拽左侧节点到画布开始编排"的中心提示
- **快捷模板**：提供常见编排模式模板（串行调用、并行调用、条件分支、循环重试）
- **画布操作提示**：底部工具栏展示缩放/平移/适配画布/对齐操作
- **智能建议**：拖入第一个节点后，自动提示"连接下一个节点"

---

### 阶段五：节点配置 & 数据映射（Configure）⭐ 核心断点

| 维度 | 内容 |
|------|------|
| **用户行为** | 选中画布上的节点 → 在配置面板中设置参数 → 配置节点间数据映射 → 设置条件分支 |
| **接触点** | 编排画布 → 右侧配置抽屉（参数面板 / 数据映射面板） |
| **情绪曲线** | 😐 专注配置 → 😫 上下文丢失 → 😤 反复切换 → 😐 完成配置 |
| **用户想法** | "上一个节点的返回值怎么传到这个节点？" "我刚才配到哪了？" |
| **当前痛点** | **画布 ↔ 配置面板切换导致上下文丢失，数据映射不直观** |
| **⚠️ 断点识别** | **核心断点**：画布与配置分离 → 上下文断裂 → 认知负担飙升 |

**优化方案：**
- **画布内嵌配置面板**：选中节点时，右侧滑出配置抽屉，**画布保持可见**（不跳转页面）
- **可视化数据映射**：连线可展开为数据映射视图，拖拽字段完成映射（Source → Target）
- **上下文锚定**：配置面板顶部显示当前节点名称 + 上下游节点关系
- **即时预览**：数据映射配置后，实时预览数据流转结构
- **配置缓存**：切换节点时，当前节点配置自动保存，回到时恢复上下文

---

### 阶段六：调试 & 测试（Debug）⭐ 核心断点

| 维度 | 内容 |
|------|------|
| **用户行为** | 运行编排流程 → 查看运行结果 → 定位失败节点 → 查看请求/响应详情 → 修改配置 → 重新运行 |
| **接触点** | 编排画布 → 运行控制 → 运行日志面板 → 节点详情 |
| **情绪曲线** | 😐 点击运行 → 😊 等待结果 → 😞 运行失败 → 😫 定位困难 → 😐 修复重试 |
| **用户想法** | "哪个节点失败了？" "请求参数对不对？" "返回了什么？" |
| **当前痛点** | **运行结果与编排画布分离，需要在多个页面间跳转才能定位问题** |
| **⚠️ 断点识别** | **核心断点**：调试需要离开编排上下文 → 断点定位效率极低 |

**优化方案：**
- **画布内状态叠加**：运行后，每个节点直接显示执行状态（✅成功 / ❌失败 / ⏳运行中 / ⏸️待运行）
- **底部测试面板**：画布底部可收起的测试面板，展示运行日志、请求/响应数据、节点执行时间线
- **失败节点高亮 + 一键定位**：失败节点红色高亮 + 点击直接展开错误详情
- **单节点独立测试**：支持选中单个节点发送测试请求，查看请求/响应
- **变量注入面板**：测试运行时可手动注入/覆盖变量值

---

### 阶段七：保存 & 发布 + 创建触发器（Deploy）⭐ 核心断点

| 维度 | 内容 |
|------|------|
| **用户行为** | 保存流程 → 发布流程 → 创建触发器 → 绑定流程 → 配置输入输出映射 → 启用触发器 |
| **接触点** | 编排画布（保存/发布） → 触发器创建页（类型选择 + 流程绑定 + 映射配置） |
| **情绪曲线** | 🙂 保存 → 😐 版本管理 → 😟 发布不确定 → 😐 触发器映射配置 → 😊 配置完成 |
| **用户想法** | "流程发布了吗？" "触发器怎么绑定？" "请求参数怎么映射到流程输入？" "流程结果怎么返回给调用方？" |
| **当前痛点** | 发布和触发器创建是两步操作，流程断裂；输入输出映射不直观 |
| **⚠️ 断点识别** | **发布→触发器两步断裂 + 输入输出映射不直观** |

**优化方案：**
- **流程发布后快捷创建触发器**：发布成功弹窗内提供"创建触发器"快捷入口，预选已发布流程
- **可视化映射配置器**：左右双列拖拽映射（Source 字段 → Target 字段），实时预览数据流
- **映射模板**：RESTful 触发器提供常见映射模板（透传、字段重命名、嵌套展开）
- **端到端测试**：触发器创建后可直接发送测试请求，验证输入→流程→输出完整链路

---

### 阶段八：运行监控 & 运维（Monitor）

| 维度 | 内容 |
|------|------|
| **用户行为** | 查看运行记录列表 → 筛选/搜索 → 查看运行详情 → 分析日志 → 告警处理 |
| **接触点** | 监控页 → 运行记录列表 → 运行详情（时间线 + 日志 + 数据流） |
| **情绪曲线** | 😐 查看记录 → 😊 快速定位 → 😟 发现异常 → 🙂 处理完成 |
| **用户想法** | "最近有哪些失败？" "具体哪个接口报错了？" "响应时间正常吗？" |
| **当前痛点** | 日志信息分散、缺乏时间线视图、告警不及时 |
| **⚠️ 断点识别** | **日志分散 + 缺乏可观测性** |

**优化方案：**
- **运行记录仪表盘**：成功率趋势图 + 失败排行 + 平均响应时间
- **可视化时间线**：运行详情页展示节点执行时间线（甘特图样式），快速定位慢节点
- **结构化日志**：按节点分组展示日志，支持关键词搜索和级别筛选
- **告警通知**：失败时即时通知（应用内通知 + 可选 webhook/邮件）

## 6. 核心设计原则（从旅程推导）

基于用户旅程分析，推导出以下 UX 设计原则：

### 原则 1：画布即编辑器（Canvas as Editor）
> **核心理念**：在编排画布中完成所有操作，永不离开编排上下文。

- 节点配置以侧边抽屉形式内嵌在画布中
- 测试结果以底部面板形式内嵌在画布中
- 数据映射以连线交互形式在画布中完成

### 原则 2：渐进式信息披露（Progressive Disclosure）
> **核心理念**：复杂配置按需展开，避免一次性信息过载。

- 资源配置表单根据类型动态展示对应字段
- 工具参数配置分 Tab 展示：基础 → 输入 → 输出 → 高级
- 节点配置面板折叠分组：必填字段默认展开，高级选项可折叠
- 触发器映射配置分步引导：绑定流程 → 输入映射 → 输出映射 → 错误处理

### 原则 3：状态始终可见（Always Visible State）
> **核心理念**：系统状态、操作结果、变更历史，始终清晰可感知。

- 资源列表显示连接状态（🟢正常 / 🟡慢 / 🔴异常）
- 工具/流程列表显示发布状态（🟡草稿 / ✅已发布 / ⬇已下线）
- 画布中每个节点实时显示运行状态
- 保存状态指示器始终在视口顶部
- 版本号和未保存变更计数始终可见

### 原则 4：快速反馈循环（Rapid Feedback Loop）
> **核心理念**：操作后 300ms 内给予反馈，错误信息紧贴操作位置。

- 表单字段实时校验（输入即校验）
- 保存操作即时反馈（保存中 → 已保存 → 保存失败）
- 节点拖放即时吸附 + 对齐辅助线

### 原则 5：一致交互模式（Consistent Interaction）
> **核心理念**：统一的操作方式降低学习成本。

- 所有节点配置面板使用相同的布局结构
- 所有列表页使用相同的筛选/排序/分页模式
- 批量导入向导统一使用三步流程（选择方式 → 上传 → 预览确认）
- 统一的右键菜单和快捷键体系

### 原则 6：发布链路连贯（Connected Publish Chain）
> **核心理念**：工具→流程→触发器的发布链路保持连贯，减少断裂。

- 工具发布后提示"在流程中使用"
- 流程发布后提供"创建触发器"快捷入口，预选已发布流程
- 触发器创建后可直接端到端测试（发送请求→流程执行→返回结果）
- 运行记录可一键跳转到对应的触发器、流程、工具进行修复

## 8. 设计风格建议

基于 ui-ux-pro-max 研究结果，推荐采用 **专业简洁 + 深色优先** 的设计风格：

| 维度 | 建议 |
|------|------|
| **配色** | 深色背景 `#0F172A` / `#1E293B`，主色调 `#6366F1`（Indigo），成功绿 `#22C55E`，失败红 `#EF4444`，警告黄 `#F59E0B` |
| **字体** | 标题/UI：`IBM Plex Sans`，代码/数据：`JetBrains Mono` |
| **布局** | 信息密度适中，核心操作区域最大化，辅助信息可折叠 |
| **动效** | 150-300ms 过渡，尊重 `prefers-reduced-motion` |
| **主题** | 默认深色，支持浅色/系统主题切换 |
| **图标** | Lucide Icons（SVG），禁止使用 emoji 作为 UI 图标 |
